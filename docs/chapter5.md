第5章 功能验证与测试

5.1 验证环境搭建

NeoRV32 的验证体系建立在 riscv-arch-test 测试框架之上。该框架由 RISC-V 国际基金会维护，提供了一套标准的指令集兼容性测试套件，用于验证处理器实现与 RISC-V 指令集架构规范的符合性。

验证环境的搭建首先需要完成 riscv-arch-test 的集成。该测试框架采用 YAML 配置文件描述被测设备（DUT）的特性，包括编译工具链路径、参考模型可执行文件路径以及 DUT 特定的宏定义文件。NeoRV32 的配置文件指定了 RV32I 指令集为测试目标，同时排除了特权架构测试项。

Verilator 仿真平台作为验证的核心执行引擎，负责将 Chisel 生成的 Verilog 代码编译为可执行仿真模型。Verilator 将 RTL 代码转换为高度优化的 C++ 模型，相比传统事件驱动仿真器在仿真速度上具有数量级的优势。仿真平台通过 C++ 测试平台（Testbench）实现时钟生成、复位控制以及与仿真模型的接口交互。

自动化测试脚本 run.py 协调整个验证流程的执行。该脚本首先调用 riscv-arch-test 框架生成测试 ELF 文件，随后提取关键符号地址（如测试签名区的起始与结束地址），将 ELF 转换为 Verilog 十六进制格式，最后编译仿真模型并逐条执行测试用例。脚本支持批量执行与结果汇总，显著提升了回归测试的效率。

5.2 单元测试

除 riscv-arch-test 架构级测试外，项目还实现了基于 Chisel 的模块级单元测试。单元测试针对流水线各级与功能单元独立验证其正确性，采用 Chisel 测试框架与 ScalaTest 断言库实现。

单元测试覆盖关键功能模块，包括取指阶段（FetchStage）的程序计数器更新逻辑、译码阶段（DecodeStage）的控制信号生成、执行阶段（ExecuteStage）的分支跳转计算以及算术逻辑单元（ALU）的运算功能。以跳转分支指令测试为例，测试用例验证跳转并链接（JAL）指令的目标地址计算、跳转并链接寄存器（JALR）指令的最低位清零特性，以及条件分支指令（BEQ、BNE 等）的跳转判断逻辑。

寄存器堆的读写测试验证异步读端口与同步写端口的正确性，特别是 x0 寄存器恒零特性的实现。此类单元测试在 RTL 级别快速定位功能缺陷，减少了全系统仿真调试的复杂度。

5.3 指令集兼容性测试

指令集兼容性是 NeoRV32 验证工作的核心目标。riscv-arch-test 框架针对 RV32I 基础指令集的每条指令提供了专用测试程序，覆盖操作数边界条件、异常行为以及指令间交互场景。

测试执行流程遵循标准化模式。每条测试程序在执行完成后，将预定格式的签名数据写入内存中的签名区。仿真平台监测签名区的写入操作，在仿真结束时提取签名数据并与参考模型的预期输出进行比对。签名比对机制确保了处理器行为与 RISC-V 规范的一致性。

边界条件测试针对指令集的特殊情形进行验证，包括分支指令的延迟槽处理、跳转目标地址的对齐要求以及访存操作的对齐限制。NeoRV32 当前版本假设所有访存操作均按字对齐，测试框架中相应跳过非对齐访存测试用例。

5.4 多配置回归测试

参数化设计的验证要求确保各配置变体的功能等价性。回归测试覆盖单周期、三级流水与五级流水三种流水线配置，验证相同程序在不同配置下的执行结果一致性。

回归测试的挑战在于流水线深度差异导致的时序行为变化。单周期配置中指令执行周期固定，而流水线配置中分支指令与数据相关可能引入停顿周期。测试框架通过仅比较最终签名数据而非执行周期数，确保功能正确性不受实现细节影响。

配置切换的测试验证参数化机制的正确性，包括隐式配置对象的传递、条件生成模块的包含与排除以及总线接口的一致性。各预定义配置（E1T、IM3UG、I5UT）均通过完整的 riscv-arch-test 测试套件验证，确保参数化设计未引入功能性缺陷。
